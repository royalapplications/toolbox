## Keeper Secrets Manager Dynamic Folder

This Dynamic Folder imports credentials from Keeper Security using the **Keeper Secrets Manager (KSM) SDK**. This Python implementation works on both **Windows and macOS**.

Items are imported as Dynamic Credentials. The username and password fields remain empty after reloading and are only fetched when a connection is established.

### Requirements

- Royal TS v7 or higher / Royal TSX v6 or higher
- Python 3.9 or higher
- Keeper Secrets Manager SDK: `pip install keeper-secrets-manager-core`
- Keeper Business or Enterprise account with Secrets Manager enabled
- KSM Configuration file (.ini, .json, or .base64 from Keeper Vault)

### Configuration

Set the **KSM Config Path** custom property to point to your config file. Supported formats:

- `.json` - From Keeper Vault → Secrets Manager → Add Device → Configuration File → JSON
- `.base64` - From Keeper Vault → Secrets Manager → Add Device → Configuration File → Base64
- `.ini` - From Keeper CLI: create KSM app, get one-time token, run `ksm init default --token YOUR_TOKEN`

All config formats work on both **Windows (Royal TS)** and **macOS (Royal TSX)**. Use the platform-specific path fields (KSM Config Path for Windows / macOS).

### Supported Record Types

- `login` - Standard username/password credentials
- `serverCredentials` - Server credentials with host/port
- `databaseCredentials` - Database credentials (MySQL, PostgreSQL, MSSQL, Oracle, MongoDB, Redis; Windows: External Application; macOS: Terminal/iTerm2 with helper script)
- `sshKeys` - SSH private keys (KeyFileContent supported)

For `databaseCredentials`: Supports **MySQL**, **PostgreSQL**, **MSSQL**, **Oracle**, **MongoDB**, and **Redis**. Database type is read from the Keeper record's Type field. **Windows** uses External Application; **macOS** uses Terminal (iTerm2-compatible) with a helper script. Set each database client path you use and KSM Config Path in Custom Properties. On macOS, the helper script is written to `~/.config/royal-keeper/db_connect.py` (launcher and helper use safe quoting and atomic writes; database passwords are passed via environment variables or short-lived local files where the client supports it). Credentials are not written to the Royal document or script logs.

### Security notes

- **Windows External Application** builds command lines using Royal credential tokens (`$$EffectivePassword$$`, etc.). After Royal substitutes values, secrets may be visible in process listings like any other launched app—this is a limitation of the External Application model, not unique to this folder. Additionally, passwords containing special characters such as `&`, `%`, `"`, or `!` may break command-line parsing and cause authentication failures. If you encounter this, consider updating the Keeper record password to avoid these characters for database connections launched via External Application.
- **macOS** launcher scripts quote all paths and arguments (`shlex`) to avoid shell injection from malicious paths in Custom Properties.
- **.base64** KSM configs are decoded to a restrictive (0600) temp file during each script run and removed when the run finishes.

### Usage

1. Reload the Dynamic Folder to see your Keeper credentials
2. Create a new connection (SSH, RDP, SFTP, etc.)
3. Enter the host and port manually
4. In the Credentials tab, select "Specify a credential name"
5. Choose your Keeper credential from the dropdown
6. Connect - credentials are fetched securely from Keeper
